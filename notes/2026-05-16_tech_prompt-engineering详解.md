# Prompt Engineering 体系详解

> 2026-05-16 | tech | Hermes Agent 讲解

---

## 一、问题起源：提示词散落导致维护灾难

之前的状态：
- `rag/chain.py` 里有 RAG 系统提示词
- `sql/agent.py` 里有 SQL Agent 提示词
- `sql/tools.py` 里有工具描述

改一个字要在三个文件里翻，面试官问"你的提示词怎么设计的"说不清楚。

类比：就像数据仓库的 SQL 脚本散落在 DataStage Job、Kettle 转换、Shell 脚本里——改一个业务逻辑要跨三个工具找。

---

## 二、结构化 Prompt 体系

### 三层设计

```
prompts/
├── system.py      ← 系统级: 角色设定 + 能力边界（管全局）
├── rag.py         ← 模块级: RAG 专用提示词（管一个领域）
├── sql.py         ← 模块级: SQL 专用提示词
└── examples.py    ← 任务级: Few-shot 示例（管具体输出格式）
```

每一层独立修改、独立测试，不会互相污染。

### 类比数据分层

| 数据分层 | Prompt 分层 |
|---------|------------|
| ODS（原始数据层） | system.py（基础角色定义） |
| DWD（明细数据层） | rag.py / sql.py（领域提示词） |
| ADS（应用数据层） | examples.py（Few-shot 应用示例） |

---

## 三、Few-shot 为什么有效

LLM 本质是"模式匹配器"——你给它看 3 个例子，它就模仿格式。这和 ETL 里的参数化模板一模一样。

```
示例4 — 排行查询：
用户：销售额最高的3个产品是哪些？
助手：[SQL] SELECT p.name, SUM(s.amount)... 
      [结果] 销售额最高的3个产品：1. 企业级数据分析平台（¥456,800）...

→ LLM 学会了：回答数据问题要展示 [SQL] 标签
```

---

## 四、选型对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **文件分层** ✅ | 改一个模块不影响其他，面试讲得清楚 | 需要约定导入规范 |
| 全写在一个大文件 | 简单 | 越改越乱 |
| 硬编码在各 .py 里 | 零额外文件 | 散落、不可维护 |

---

## 五、设计要点

1. **系统级 (system.py)**：MAIN_SYSTEM_PROMPT + CAPABILITY_STATEMENT + OUTPUT_FORMAT_RULES
2. **模块级 (rag.py / sql.py)**：具体领域提示词
3. **统一导出 (\_\_init\_\_.py)**：所有 import 集中管理，外部只需 `from prompts import X`

---

## 六、常见坑

1. **迁移时丢格式** — 多行字符串迁移时容易丢缩进，保持原样
2. **循环导入** — prompts 是底层，只被引用，不引用别人
3. **过度设计** — 提示词 3 层就够了，不要搞 5-6 层

---

## 七、面试话术

### 30秒版
"我把项目中的所有提示词从散落在各个 .py 文件里，重构到 prompts/ 目录下，按系统级、模块级、任务级三层管理。每个模块的提示词独立可测试，面试中我能讲清楚每一层为什么这么设计。"

### 3分钟版
"传统做法是把提示词硬编码在业务代码里，改一个字要翻三个文件。我参考了 Prompt Engineering 的最佳实践，做了三层分离：系统级定义角色和能力边界，模块级定义 RAG 和 SQL 各自的领域提示词，任务级用 Few-shot 示例控制输出格式。核心思想是让 Prompt 从"散落的字符串"变成"可管理的工程资产"。这和数据库的分层建模 ODS/DWD/ADS 是一个道理。"
