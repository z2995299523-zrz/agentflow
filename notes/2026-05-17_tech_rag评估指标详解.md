## 一、四个自实现指标

### 1. Recall@5 — 找得全不全

**公式：** `命中的相关文档 / 总相关文档`

**含义：** 知识库里有 3 个文档提到"年假"，检索 Top-5 返回了其中 2 个 → Recall = 2/3 = 67%。

**为什么 @5？** 用户最多看前 5 个结果（Google 翻到第二页的概率 <10%），只评估 Top-5 覆盖度。

**ETL 类比：** 数据完整率。源表 100 万行，目标表 99 万行 → 完整性 99%。Recall 也是完整性——相关 3 篇，找到 2 篇 → 完整 67%。

**我的基线：90%。** 10 题 9 题命中，BGE-small-zh 中文语义理解好。

**面试话术：**
> "Recall@5 衡量检索召回率——相关文档在 Top-5 里出现了多少。基线 90%，意味着 10 题 9 题能在前 5 个结果里找到正确答案。"

---

### 2. Precision@5 — 找得准不准

**公式：** `命中的相关文档 / 返回的 5 个结果`

**含义：** 返回 5 个结果，3 个相关，2 个不相关 → Precision = 3/5 = 60%。

**与 Recall 的跷跷板关系：**

| 策略 | Recall | Precision | 行为 |
|------|--------|-----------|------|
| 宽松检索 | 高 | 低 | 什么都捞，用户自己筛 |
| 严格检索 | 低 | 高 | 只给最确定的，可能漏 |
| **平衡（目标）** | ≥80% | ≥60% | 兼顾 |

**ETL 类比：** 数据准确率。源表 1000 行，清洗后 800 行有效 → 准确率 80%。

**我的基线：30%。** 测试集只有 4 个文档，k=5 时几乎全返回，分母永远是 3-4 个文档、分子是 1 个 → 1/3 ≈ 33%。这不是检索质量问题，是数学必然。生产环境有上百文档时自然提上去。

**面试话术：**
> "Precision 随文档量增长自然提升。4 个文档的测试集 k=5 必然全返回。真实上百个文档时，BGE 语义排序把不相关的排后面，Top-5 的 Precision 自然上去。"

---

### 3. MRR — 第一个对的排第几

**公式：** `1 / 第一个相关结果的排名`

**含义：** 排第 1 位 → MRR=1.0。排第 3 位 → 0.33。排第 10 位 → 0.10。没找到 → 0。

**为什么重要？** 用户只看第一个结果。MRR 衡量"正确答案离用户的眼球有多近"。

**ETL 类比：** 关键数据到达率。监管报表 T+1 早上 8 点前必须到——8:01 算迟到。MRR 也一样：排第 2 = 迟到，排第 1 = 准时。

**我的基线：0.90。** 9 题正确答案排第 1（MRR=1.0），1 题"今天天气"无相关文档（MRR=0），平均 0.9。

**面试话术：**
> "MRR 衡量第一个正确答案的排名。0.90 意味着 90% 问题正确答案排第一位。这个指标对用户体验影响最大——没人看第二个结果。"

---

### 4. Answer Relevance — 回答靠不靠谱

**公式（简化版）：** `答案中包含标准答案关键词的比例`

**含义：** 回答是否基于检索内容？还是 LLM 在编造？

**ETL 类比：** 数据可信度。报表每个数字都追溯到源表的某一行。RAG 也一样——回答每句话都应该能在检索文档里找到出处。

**我的基线：70%。** 两道题没达标，"周末加班薪资"和"加密方式"——LLM 用自己的知识回答了，没有严格基于检索内容。这是 RAG 经典问题：LLM 太聪明容易"越狱"。

## 二、RAGAS Faithfulness — 逐句溯源

### 问题起源

Answer Relevance 看的是"回答有没有提到标准答案关键词"——太粗糙了。

举个例子：
- 问题："年假有多少天？"
- 检索内容："入职满1年享有5天年假"
- LLM 回答："入职满1年享有5天年假，满3年10天，满5年15天"

Answer Relevance 会说 100%——因为答案包含"5天"。但"满3年10天，满5年15天"是从哪来的？检索内容里只说了"满1年5天"。LLM **编造了后半句**！

这就是 **Faithfulness（忠实度）** 要解决的问题。

### 核心原理

Faithfulness 不看"答案和标准答案像不像"，而是看"答案里的每句话能不能从检索内容里找到出处"。

**算法流程：**

```
LLM 回答: "入职满1年享有5天年假，满3年10天，满5年15天。年假需提前申请。"
    │
    ▼ 第1步：拆成句子（用 LLM 或标点分割）
┌─────────────────────────────────────┐
│ 句子1: "入职满1年享有5天年假"        │
│ 句子2: "满3年10天"                   │
│ 句子3: "满5年15天"                   │
│ 句子4: "年假需提前申请"              │
└─────────────────────────────────────┘
    │
    ▼ 第2步：每个句子和检索内容比对
┌────────────────────────────────────────────────────────┐
│ 检索内容: "入职满1年享有5天年假，需提前3个工作日申请"   │
│                                                        │
│ 句子1 vs 检索 → ✅ 能找到出处 ("入职满1年享有5天年假")   │
│ 句子2 vs 检索 → ❌ 找不到出处 (检索里没有"满3年10天")    │
│ 句子3 vs 检索 → ❌ 找不到出处 (检索里没有"满5年15天")    │
│ 句子4 vs 检索 → ✅ 能找到出处 ("年假需提前申请" ≈ "需提前3个工作日申请") │
└────────────────────────────────────────────────────────┘
    │
    ▼ 第3步：计算
Faithfulness = 可溯源的句子数 / 总句子数 = 2/4 = 50%
```

**不是简单的"包含关键词"，而是把回答拆成独立陈述，逐条查证出处的严格审计。**

### 底层实现（LLM 判定）

比对不是用字符串匹配，而是用 LLM 判断：

```python
# RAGAS 内部对每个句子做的判断
prompt = f"""
检索内容：{context}
陈述：{sentence}

这个陈述能从检索内容中推断出来吗？回答 Yes 或 No。
"""
# LLM 返回 Yes → 可信，No → 幻觉
```

**为什么用 LLM 而不是关键词匹配？** 因为"年假需提前申请"和"需提前3个工作日申请"是语义等价，关键词匹配不到"3个工作日"就不算匹配。LLM 能理解语义等价。

### ETL 深度类比

| 概念 | RAG Faithfulness | ETL 对账 |
|------|-----------------|---------|
| 拆句 | 把回答拆成独立陈述 | 把报表拆成原子字段 |
| 溯源 | 每句话找检索出处 | 每个字段找源表来源 |
| 判定 | LLM 判断语义等价 | SQL 比对值是否一致 |
| 审计结果 | 可溯源/幻觉 | 一致/差异 |
| 容错 | 语义等价算通过 | 类型转换容忍精度差异 |

你做银行数据的时候，报表上的"存款余额 1,234,567.89"要能追溯到核心系统的账户表。RAG 的 Faithfulness 完全一样——回答里的"满3年10天"要能追溯到上传的员工手册。**区别只是：你以前用 SQL JOIN 追，现在用 LLM 追。**

### 和 Recall/Precision 的区别

| 指标 | 评估对象 | 问什么 |
|------|---------|--------|
| Recall/Precision/MRR | **检索阶段** | "返回的文档对不对？" |
| Faithfulness | **生成阶段** | "LLM 有没有编造？" |

**检索准 ≠ 回答真。** 检索返回了正确文档，LLM 可能不用它，用自己的知识回答。Faithfulness 就是抓这个的。

### 面试话术（30秒版）

> "Faithfulness 是 RAGAS 框架的核心指标，衡量 LLM 回答对检索内容的忠实度。把回答拆成句子，逐句和检索内容比对，计算能追溯的比例。它解决的是 RAG 的头号问题——LLM 幻觉。"

### 面试话术（3分钟深度版）

> "Faithfulness 和传统指标有本质区别。Recall 和 Precision 评估的是检索阶段——文档找得对不对。但检索对 ≠ 回答对。
>
> RAG 最致命的问题是 LLM 幻觉——检索给了正确文档，LLM 不用，用自己的参数化知识回答。比如检索说'入职满1年5天年假'，LLM 回答'入职满1年5天，满3年10天，满5年15天'——后半句是编的。
>
> Faithfulness 的做法是把回答拆成原子陈述，每个陈述和检索内容做语义比对，用 LLM 判断是否能从检索内容推断。可溯源的陈述 / 总陈述 = Faithfulness 分数。
>
> 这和我以前做 ETL 数据对账的思路完全一样——报表的每个数字都要能追溯到源表的某一行。区别只是以前用 SQL JOIN，现在用 LLM 判断语义等价。"

---

### 优化路线（按谁低优化谁）

| 指标低 | 问题定位 | 优化方案 |
|--------|---------|---------|
| Recall 低 | 检索没覆盖 | 混合检索（BM25 关键词 + 向量） |
| Precision 低 | 返回太多无关 | 重排序（BGE-Reranker） |
| MRR 低 | 正确答案排后面 | 重排序 + Query 改写 |
| Faithfulness 低 | LLM 编造 | 加强 Prompt 约束 + 降低 temperature |

**原则：先测基线，拿数字，再优化。没有 baseline 的优化是玄学。**
